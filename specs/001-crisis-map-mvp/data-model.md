# Data Model: CrisisMap MVP

**Feature**: 001-crisis-map-mvp | **Date**: 2026-06-27

Storage is DynamoDB with three simple tables (per the constitution). This
document defines entities, attributes, keys, indexes, validation rules, and
state transitions. Types are shown as DynamoDB attribute types (S = string,
N = number, B = binary, L = list, M = map).

---

## Enums (shared across entities)

### IncidentType (closed enum, 20 values)
`people_trapped`, `building_collapse`, `damaged_building`, `fire`, `flood`,
`road_blocked`, `bridge_damaged`, `landslide`, `hospital`, `shelter`, `food`,
`water`, `medicine`, `electricity`, `fuel`, `internet`, `starlink`,
`open_wifi`, `charging_point`, `other`.

### IncidentCategory (derived from type, drives marker colour)
- `emergency` (red): people_trapped, building_collapse, fire, flood,
  road_blocked, bridge_damaged, landslide, damaged_building
- `infrastructure` (orange): electricity
- `service_interruption` (yellow): fuel
- `resource` (green): hospital, shelter, food, water, medicine
- `communications` (blue): internet, starlink, open_wifi, charging_point
- `other` maps to `emergency` colour by default (configurable).

### Severity (closed enum, 3 values)
`low`, `medium`, `high`.

### IncidentStatus (closed enum)
`active`, `resolved`.

### ConfirmationAction (closed enum, 4 values)
`confirm`, `improved`, `worsened`, `no_longer_exists`.

## Table: Incidents

One row per reported incident or resource.

| Attribute | Type | Key | Notes |
|-----------|------|-----|-------|
| `incidentId` | S | PK | UUID v4, generated by backend on creation. |
| `geohash` | S | GSI1 PK | base32, precision 7 (~153 m cell). |
| `type` | S | GSI2 PK | one of IncidentType. |
| `location` | M | — | `{ lat: N, lng: N }`. |
| `severity` | S | — | `low` \| `medium` \| `high`. |
| `status` | S | — | `active` \| `resolved`. Default `active`. |
| `description` | S | — | free text, max 500 chars, optional. |
| `createdAt` | N | GSI1 SK | epoch seconds. |
| `updatedAt` | N | — | epoch seconds. |
| `expiresAt` | N | (TTL attr) | epoch seconds; `createdAt + 72h`, reset to `now + 72h` on each confirmation. |
| `creatorAlias` | S | — | alias at creation time; optional. |
| `creatorDeviceId` | S | — | the device that created it. |
| `confirmations` | N | — | count; starts at 1 (creator's implicit confirmation). |
| `negativeVotes` | N | — | count; starts at 0. |
| `imageCount` | N | — | number of attached images; 0–3. |

**Keys & indexes**
- Partition key: `incidentId`.
- GSI1 (`geohash`, `createdAt`): bbox cell queries (newest first within a
  cell).
- GSI2 (`type`, `geohash`): duplicate detection (same type in nearby cells) and
  type filters.

**Validation rules**
- `type` MUST be in IncidentType.
- `severity` MUST be in Severity.
- `confirmations >= 1` always (creator's implicit confirmation).
- `negativeVotes >= 0`; displayed confidence = `max(0, confirmations - negativeVotes)`.
- `imageCount` MUST be 0–3.
- `description` length <= 500 chars (truncated if exceeded).
- `location.lat` in [-90, 90], `location.lng` in [-180, 180].

**TTL / visibility**
- DynamoDB TTL set on `expiresAt` as an eventual-cleanup backstop only.
- Default-view filtering excludes `expiresAt < now` OR `status = resolved`
  in application logic (never relies on TTL for correctness, to honour "do not
  delete").

## Table: Confirmations

One row per (incident, device) verification action. Enforces one-action-per-
device at the partition level.

| Attribute | Type | Key | Notes |
|-----------|------|-----|-------|
| `incidentId` | S | PK | references Incidents.incidentId. |
| `deviceId` | S | SK | the acting device. |
| `action` | S | — | one of ConfirmationAction. |
| `createdAt` | N | — | epoch seconds. |

**Keys & indexes**
- Partition key: `incidentId`, sort key: `deviceId`.
- Composite key `(incidentId, deviceId)` is unique — a second action from the
  same device is a conditional-write failure (409).

**Side effects on apply** (within the same Lambda transaction or follow-up
update, applied idempotently):
- `confirm` → Incidents.confirmations += 1, Incidents.expiresAt = now + 72h.
- `improved` → Incidents.confirmations += 1, Incidents.expiresAt = now + 72h.
- `worsened` → Incidents.negativeVotes += 1.
- `no_longer_exists` → Incidents.status = `resolved`.

**Validation**
- `action` MUST be in ConfirmationAction.
- `deviceId` MUST be non-empty (client must register a deviceId before
  confirming).

## Table: Devices

Lightweight registry of a device's identity. Written once on first interaction;
alias updates overwrite in place.

| Attribute | Type | Key | Notes |
|-----------|------|-----|-------|
| `deviceId` | S | PK | UUID v4 generated client-side. |
| `alias` | S | — | optional, changeable; may be empty. |
| `createdAt` | N | — | epoch seconds. |

**Keys & indexes**: partition key `deviceId` only.

**Validation**: `deviceId` is a non-empty UUID; `alias` length <= 30 chars.

---

## Client-side store (IndexedDB)

Not a server table — the local persistence layer for offline-first behaviour.

| Store | Contents |
|-------|----------|
| `incidents` | cached incidents fetched for viewed bbox (key: `incidentId`). |
| `outbox` | pending operations `{ id, op, payload, createdAt }` awaiting sync. |
| `device` | single record `{ deviceId, alias }`. |

Outbox operation types: `create_incident`, `confirm`, `upload_image`.
Sync drains the outbox in batches and removes each op on success.

## State transitions

### Incident status
```
                  POST /incidents (new)
                           |
                           v
                        active
                           |
            no_longer_exists (ConfirmationAction)
                           |
                           v
                        resolved
```
- `active` → `resolved`: when any device sends `no_longer_exists`.
- `resolved` → `active`: not supported (a resolved incident stays resolved).
- There is no `deleted` state; expired/resolved incidents are filtered from the
  default view but never removed.

### Incident visibility (derived, not stored state)
- **Visible in default view**: `status = active` AND `expiresAt >= now`.
- **Hidden (default view)**: `status = resolved` OR `expiresAt < now`.
- **Visible via explicit filter**: any status/expiration (filters can opt to
  show hidden items).

### Confirmation lifecycle per device
- A device starts with NO record against an incident.
- First action creates the `(incidentId, deviceId)` row and applies side
  effects.
- Any further action from the same device is rejected (409) — no state change.

## Counts and confidence (derived)

- `confidence = max(0, confirmations - negativeVotes)`.
- Display text: `Confirmed by {confirmations} people` (or "Confirmed by X
  people" where X = confirmations).
- The "Confirmed only" client filter shows incidents where `confirmations >= 2`
  (at least one other person besides the creator).
